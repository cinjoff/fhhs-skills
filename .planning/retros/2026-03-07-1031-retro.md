### Discoveries

- **GSD binary is just `bin/gsd-tools.cjs` + `bin/lib/*.cjs`** — the runtime dependency is purely filesystem, no npm install needed. Symlink is sufficient for backwards compatibility
- **The simplify skill from Superpowers is baked in at the skills level** — it's not a plugin, it's part of the Superpowers plugin's native skill registry. This is why `/skill` invocations can find it without it being in fhhs-skills
- **The project is explicitly TS/JS-opinionated** — this should be a standing assumption, not something to ask about case by case

### Friction Points

- **Scope creep without explicit scoping** — the conversation evolved from "add /simplify to build" → "add LSP to skills" → "review everything" → "internalize /simplify" → "make LSP mandatory" → "fully internalize GSD". Each pivot was reasonable but accumulated unplanned work. Explicit scope confirmation earlier would help
- **GSD internalization was ambitious** — the symlink + bundling approach required significant exploration (10+ bash commands) to understand the runtime structure before proceeding. Asking whether the user wanted a quick path vs full solution upfront would have helped
- **Multiple passes over same files** — skills-guide.md was edited 6+ times across the session as each new decision rippled through. A single consolidation pass at the end would have been more efficient

### Actionable Learnings

1. **Always check if a referenced skill/tool is external or internal before integrating it** — when adding `/simplify`, the first question should have been: "Is this in fhhs-skills or a Superpowers dependency?"
2. **Save project stack assumptions to memory** — "This project is TS/JS-opinionated. LSP is always available and mandatory, not optional." Add this to `MEMORY.md` so it doesn't get recategorized as optional again
3. **When adding cross-cutting concerns (LSP, simplify), check all integration points first, then edit in one pass** — finding all files before editing prevents the 6-edit-per-file pattern
4. **The GSD symlink pattern is now established** — `$HOME/.claude/get-shit-done/bin/` → plugin's `bin/`. Document this in memory so future update sessions don't re-investigate the structure
5. **For sessions likely to span many files, start with a TodoWrite list** — the scope here was large enough that tracking completed vs pending edits explicitly would have reduced the "did I update that file?" re-reads
