  <name>Task 2: Make release command portable + add health checks</name>
  <files>.claude/commands/release.md</files>
  <action>
  Read the current release.md. Make two changes:

  **2a. Fix hardcoded paths:**
  Replace all occurrences of `cd /Users/konstantin/Documents/github.nosync/fhhs-skills` with:
  ```bash
  REPO_ROOT=$(git rev-parse --show-toplevel)
  cd "$REPO_ROOT"
  ```
  Since release.md is a skill (not a shell script), each bash block runs independently.
  Add `REPO_ROOT=$(git rev-parse --show-toplevel)` at the top of EVERY bash block that
  currently has the hardcoded `cd` (Steps 1, 5, and 6). Do not assume variable sharing
  between blocks.

  **2b. Add pre-ship health checks to Step 0:**
  After the existing Step 0c (quick review), add two new substeps:

  ### 0d. Eval smoke check
  ```bash
  if [ -f "fhhs-skills-workspace/run_all_evals.py" ]; then
    echo "Running eval smoke tier..."
    python3 fhhs-skills-workspace/run_all_evals.py --tier smoke 2>&1 | tail -5
  else
    echo "WARN: Eval runner not found — skipping eval check"
  fi
  ```
  **If any evals fail:** STOP and show failures. Do not proceed until fixed or user acknowledges.
  **If eval runner not found:** WARN and continue (not all environments have the eval workspace).

  ### 0e. Plugin health check
  ```bash
  SKILLS_SIZE=$(du -sh .claude/skills/ | cut -f1)
  SKILLS_COUNT=$(ls -d .claude/skills/*/ 2>/dev/null | wc -l | tr -d ' ')
  # macOS du lacks -b; use -sk (kilobytes) portably
  SKILLS_KB=$(du -sk .claude/skills/ | cut -f1)
  echo "Plugin: $SKILLS_COUNT skills, $SKILLS_SIZE total"
  if [ "$SKILLS_KB" -gt 3072 ]; then
    echo "WARN: Plugin exceeds 3MB ($SKILLS_SIZE). Consider removing unused skills."
    echo "Largest skills:"
    du -sh .claude/skills/*/ 2>/dev/null | sort -rh | head -5
  fi
  ```
  This is advisory (WARN, not STOP) — large plugins still work, but maintainer should be aware.

  Also verify shipping boundary:
  ```bash
  # Check no skill references files outside .claude/skills/
  if grep -r 'references/' .claude/skills/*/SKILL.md 2>/dev/null | grep -v 'references/gsd/' | grep -v '#'; then
    echo "WARN: Skills reference files outside shipping boundary"
  fi
  ```
  </action>

  - Step 0d and 0e substeps exist
  - release.md is valid markdown
  </verify>

<task type="auto">
  <name>Task 3: Create CONTRIBUTING.md</name>
  <files>CONTRIBUTING.md</files>
  <action>
  Create CONTRIBUTING.md at repo root with these sections:

  ## Contributing to fhhs-skills

  Brief welcome. Link to README for what the plugin does.

  ## Development Setup

  1. Clone the repo
  2. Install Claude Code
  3. `claude plugin install .` to install from local source
  4. `/fh:setup` to configure prerequisites

  ## Project Structure

  Brief description of the directory layout:
  - `.claude/skills/` — shipped skills (user-facing)
  - `.claude/commands/` — maintainer commands (not shipped)
  - `upstream/` — verbatim upstream snapshots (never edit)
  - `evals/` — eval suite
  - `bin/` — GSD CLI tooling
  - `.planning/` — project state

  ## Making Changes

  - Follow conventional commits: `feat:`, `fix:`, `docs:`, `chore:`, `refactor:`
  - Skills use `/fh:` prefix in docs
  - Runtime-read files must live inside `.claude/skills/{name}/` (shipping boundary)
  - Run evals after changes: `python3 fhhs-skills-workspace/run_all_evals.py --tier smoke`

  ## Adding a New Skill

  1. Create `.claude/skills/{name}/SKILL.md`
  2. Add frontmatter with `name`, `description`, `user-invocable` (note: with a **c**)
  3. Add at least 1 eval in `evals/`
  4. Update README.md command tables

  ## Reporting Issues

  Use GitHub issue templates for bug reports and feature requests.

  ## Release Process

  Maintainer-only. See `.claude/commands/release.md`.

  Keep it concise — under 150 lines. No badges, no verbose legal text.
