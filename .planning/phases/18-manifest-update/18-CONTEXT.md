# Phase 18: Manifest-Driven Update Infrastructure

## Decisions

- [Dedicated manifest file]: `fhhs-manifest.json` in two layers — `~/.claude/fhhs-manifest.json` (global: tools, plugins, hooks, env) and `<project>/.claude/fhhs-manifest.json` (project: stack, features, planning state). Separate from settings.json to avoid mixing concerns.
- [Single source of truth]: The manifest replaces CHANGELOG.md tags as the authority on what should exist. CHANGELOG tags remain for backward compat but the manifest is canonical.
- [Creation lifecycle]: `/fh:new-project` creates the initial project manifest (detects stack, features). `/fh:update` maintains it and backfills for existing projects that don't have one yet.
- [Script does the work]: `bin/lib/manifest.cjs` handles all detection, diffing, remediation deterministically. Every operation wrapped in try/catch, always produces valid JSON output, includes actionable remediation hints on failure. Never gets stuck — reports errors and continues.
- [Thin SKILL.md wrapper]: update/SKILL.md becomes ~60 lines. Calls the script, formats JSON output as user-friendly table, handles user decisions for optional items.
- [Auto-remove deprecated]: Manifest marks items as `deprecated` with `action: remove`. Script auto-removes them (uninstalls plugin, cleans env, removes hooks). No user confirmation needed — manifest is intentional.
- [User feedback format]: "What's new for you" changelog summary (user-centric, from CHANGELOG.md) + environment status table (component / status / details) + summary line (N removed, N fixed, N errors).
- [Project profile]: Manifest includes stack detection (framework, database, auth, testing, packageManager) so skills can make contextual decisions (e.g., don't suggest Next.js patterns for a Node CLI project).
- [Planning state tracking]: Manifest tracks `.planning/` structure presence, CLAUDE.md, conductor.json to detect stale or missing project infrastructure.
- [Resilient error handling]: Script never exits non-zero for recoverable issues. JSON output always has `status`, `errors[]` with `{code, message, hint}`, and `remediated[]`. The LLM wrapper or a human can always understand what happened.
- [review] [Absorb global-reconcile.cjs]: manifest.cjs absorbs all logic from global-reconcile.cjs (project discovery, remediation, health repair, stale file checks) and post-update-reconcile.sh (claude-mem settings, env setup, tracker registration). One engine, not two parallel systems. Both files deleted after absorption.
- [review] [changelog.cjs keeps parsing only]: Strip extractReconciliationTags(), runChecks(), cmdChangelogReconcile() from changelog.cjs. Keep parseChangelog(), compareSemver(), getEntriesBetween() for changelog display in the thin SKILL.md.
- [review] [Single project default, --all for sweep]: `manifest check` operates on current project by default. `manifest check --all` discovers and checks all projects (replaces global-reconcile's multi-project sweep). Update skill calls `--all` after version bump.
- [review] [Install command validation]: manifest-schema.cjs includes validateInstallCommand() that validates install commands match safe patterns (known installers, package name format). Rejects arbitrary shell to prevent manifest tampering.

## Discretion Areas

- [Manifest schema details]: Exact field names and nesting can be adjusted during implementation as long as the two-layer (global/project) structure is preserved.
- [Stack detection heuristics]: How framework/database/auth are detected from project files (package.json, config files) — implementer decides the detection order and confidence thresholds.
- [CHANGELOG extraction]: How "What's new for you" entries are filtered from CHANGELOG.md — can reuse existing logic or simplify.

## Deferred Ideas

- [Fire horse ASCII art]: User mentioned possible ASCII branding in output — defer to a future polish pass.
- [Manifest versioning/migration]: Schema versioning for the manifest file itself — not needed until the schema stabilizes.
- [Manifest sync across machines]: Cloud sync of global manifest — out of scope.
- [Auto-suggest capabilities]: Using project profile to proactively suggest skills/tools (e.g., "Supabase detected, want to add auth?") — future enhancement after manifest is stable.
- [review] [CLAUDE.md slimming via manifest]: Project CLAUDE.md files generated by /fh:new-project could reference the manifest for stack/testing info instead of inlining it. The repo CLAUDE.md is already thin (90 lines) — savings are marginal there but project CLAUDE.md files tend to be longer. Revisit after manifest stabilizes.
- [review] [Expansion opportunity]: The ambitious version would make the manifest a full project intelligence layer — skills query it before acting (e.g., build checks profile.testing before choosing test runner, review checks profile.framework before suggesting patterns), turning it from passive inventory into active skill routing.
