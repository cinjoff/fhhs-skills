<task type="auto" tdd="true">
  <name>Task 2: Build comparative eval harness</name>
  <files>fhhs-skills-workspace/comparative_eval.py</files>
  <action>
  Create a side-by-side eval harness that:

  1. Takes a benchmark definition (YAML):
     ```yaml
     name: "build-simple-feature"
     prompt: "Add a dark mode toggle to the settings page"
     fhhs_skill: "build"
     upstream_skill: "gsd:execute-phase"  # or superpowers equivalent
     fixture: "fixtures/react-app/"
     metrics:
       - token_usage       # total tokens consumed
       - wall_time         # seconds to complete
       - quality_score     # LLM-graded 1-10
       - files_modified    # count of files touched
       - test_coverage     # did it add tests?
     ```

  2. Runs the prompt against fhhs skill in a worktree:
     - Capture token usage from claude API response metadata
     - Measure wall time
     - After completion: run LLM grader on the output (quality 1-10 with rubric)

  3. Runs the same prompt against raw upstream skill in a separate worktree via prompt injection [review]:
     - Create worktree, inject upstream SKILL.md/PROMPT.md content as system prompt to `claude -p`
     - Same metrics capture
     - Uses the upstream snapshot markdown as the injected prompt source
     - **Known limitation [review]:** headless `claude -p` sessions lack MCP tools (claude-mem, context-mode, firecrawl), skill dispatch, and hooks. This measures "raw prompt quality" not "fully-configured environment quality." Both sides are equally limited, making comparison fair for prompt effectiveness but not representative of full plugin experience. Document this in the harness README.

  4. Produces a comparison:
     ```
     build-simple-feature:
       fhhs:     tokens=45k  time=120s  quality=8.2  files=6  tests=yes
       upstream:  tokens=62k  time=180s  quality=7.1  files=8  tests=no
       delta:     -27% tokens  -33% time  +15% quality
     ```

  5. Infrastructure:
     - Reuse existing eval fixture infrastructure (fixtures/ directory)
     - LLM grader: send both outputs to Claude with a rubric, get scores
     - Token tracking: parse from claude -p output or API response
     - Results stored as JSON for trend tracking
  </action>
  <verify>comparative_eval.py runs against a single benchmark and produces comparison output</verify>
  <done>Side-by-side eval harness working with token, time, and quality metrics</done>
</task>
