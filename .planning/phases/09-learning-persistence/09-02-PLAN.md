  3. Add new Section 3.5 "Project Insights" between existing Section 3 (Insights Dashboard) and Section 4 (File GitHub Issues). This section:

     a. **Classify from existing working set** — Do NOT make additional claude-mem API calls. Instead, classify observations already collected in the **working set** (Section 2c) by scanning each observation's text for category keywords:
        - **architecture**: "architecture", "decision", "component", "structure", "pattern", "design", "boundary", "coupling"
        - **testing**: "test", "coverage", "bug", "regression", "broken", "fix", "failing", "flaky"
        - **workflow**: "slow", "blocked", "stuck", "retry", "workaround", "timeout", "token", "expensive"
        - **tech-debt**: "debt", "cleanup", "refactor", "duplication", "hack", "temporary", "todo", "fixme"
        - **scope**: "scope", "creep", "large", "split", "took long", "estimate", "phase", "overran"

        An observation can match multiple categories. Assign to the strongest match (most keyword hits).

     b. **Pattern analysis** — For each category, analyze the classified observations looking for:
        - **architecture**: Decisions that led to repeated issues or smooth execution. Patterns in which areas of the codebase generate the most bugs/changes.
        - **testing**: Files or features that keep breaking, suggesting missing test coverage. Repeated debugging cycles.
        - **workflow**: Steps that consistently take longer or require retries. Tool usage patterns that waste tokens.
        - **tech-debt**: Areas where workarounds accumulate. Files modified most frequently (churn).
        - **scope**: Phases that exceeded estimates. Features that kept expanding during implementation.

     c. **Generate insights** — For each category with sufficient signal (2+ observations), produce an insight with:
        - Category tag (architecture, testing, workflow, tech-debt, scope)
        - Observation summary (what the pattern is)
        - Evidence count and date range
        - **Concrete suggested action** — not "consider improving tests" but "The auth module had 4 bug fixes in 14 days. Add integration tests covering login, signup, and token refresh flows."

     d. **Present as:**
        ```
        ## Project Insights

        Patterns detected from your recent work that suggest improvement opportunities:

        | # | Category | Pattern | Evidence | Suggested Action |
        |---|----------|---------|----------|-----------------|
        | 1 | testing | Auth module: 4 bug fixes in 14 days | 4 obs | Add integration tests for login, signup, token refresh |
        | 2 | scope | Phase 3 took 2x longer than phases 1-2 | 6 obs | Consider splitting phases with >8 tasks |
        | ... | | | | |

        ### Detailed Recommendations

        For each insight, expand with:
        - **What we observed:** [specific observation quotes]
        - **Why this matters:** [impact on velocity/quality/cost]
        - **Suggested action:** [concrete, actionable steps]
        - **Priority:** high (3+ recurring) / medium (pattern emerging) / low (single signal)
        ```

     e. If no project insights are found (fewer than 2 observations in all categories), say: "No strong patterns detected in the last {N} days. This is normal for new projects or short time windows."

          - For **architecture** insights: "Refactor [area] to address [recurring pattern]"
          - For **workflow** insights: "Optimize [step] that caused repeated issues"
          - For **tech-debt** insights: "Clean up [area] where workarounds accumulated"
          - For **scope** insights: "Split [large area] into smaller phases"
       3. Present the generated prompt to the user for confirmation, then invoke `/fh:plan-work` with it

       If the user replies 'skip', proceed to Section 5.

       **If NO GSD project**, present a structured improvement brief instead:

       > **Top improvements to consider:**
       >
       > 1. [Highest priority insight — category, pattern, suggested action]
       > 2. [Second priority insight]
       > 3. [Third priority insight]
       >
       > To turn these into an execution plan, set up project tracking first with `/fh:new-project`, then run `/fh:learnings` again.

  5. Update Section 5 (Offer Extended Scope) to also mention: "You can also run `/fh:learnings` after your next phase to track how patterns evolve over time."
